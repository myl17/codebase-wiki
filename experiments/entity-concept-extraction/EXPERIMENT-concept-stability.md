# 实验：Ontology 锚定对 Concept 提取稳定性的影响

> 设计日期：2026-06-20
> 核心假设：给定已聚合的 Ontology 归属，Concept 提取的 inter-run 稳定性显著高于没有 Ontology 时的旧流程。

---

## 一、实验管线总览

```
Phase 1: 每个仓库独立提取 Structural Entity + 候选 Ontology 节点
  ↓         （独立 subagent，独立上下文，不知道其他仓库）
Phase 2: 跨仓库 Ontology 对齐
  ↓         （将独立提取的 Ontology 节点做跨仓库匹配和归并）
Phase 3a: 域内 Concept 提取
  ↓         （给定 Ontology 节点下已聚合的跨仓库实例，提取设计知识）
Phase 3b: Concept 独立验证
            （源码验证每个 Concept 页的 claim）
```

**增量执行**：

| Round | 涉及仓库 | Concept 提取 |
|-------|---------|-------------|
| R1 | openclaw（Phase 1 only） | 不可生成（< 2 仓库） |
| R2 | openclaw + hermes | 对共同 Ontology 节点提取 Concept × 3 次 |
| R3 | openclaw + hermes + nanobot | 对更新后的 Concept 重新提取 × 3 次 |

## 二、Phase 1: Structural Entity + Ontology 提取

### 提示词模板（通用，不包含仓库特定信息）

```
你的任务是直接从 <仓库名> 源码中提取 Structural Entity 和候选 Ontology 节点。

## 输入源

<仓库名> 源码在 <源码路径>。
这是你的唯一信息来源。不用训练数据中可能存在的知识。

## 工作方式

你自己决定读哪些文件。广泛探索直到全面理解这个仓库的结构。

## 第一步：提取 Structural Entity

对每个 Entity，你需要产出的是一个**有技术深度的页面**——不是只标注文件路径。格式：

### Entity: <名称>

**代码位置**：<目录/包路径>
**这个模块解决什么问题**：<一句话>
**对外暴露什么**：<关键类/函数/接口，含文件路径:行号>
**它和谁交互**：<依赖的其他模块/被谁调用>
**为什么它是可分离的**：<独立目录？独立接口？独立的包？>

**关键机制**（源码可见）：
- <机制 1>：<描述> ^[文件路径:行号]
- <机制 2>：...

**源码证据**：
- 入口文件：<路径>
- 核心类型/接口定义：<路径:行号>

## 第二步：提取候选 Ontology 节点

Ontology 节点 = 跨仓库可对齐的"能力域"。但你**不要凭空创造分组**——只标注原始开发者已经通过代码结构表达出来的分组。

判断标准（**必须基于源码证据，不能凭空判断**）：
1. **接口/抽象类 + 多个实现**：源码中存在一个 interface/ABC，后面跟着 ≥2 个 impl？→ 候选 Ontology 节点
2. **独立的包/目录 + 对外的类型/函数导出**：存在独立目录，有明确的入口文件，导出类型/接口？→ 候选 Ontology 节点
3. **注册机制**：存在 register*()、add_*() 等显式注册函数，允许外部添加新实现？→ 候选 Ontology 节点
4. **配置项中的可替换策略**：存在配置项允许用户在多个策略间切换？→ 候选 Ontology 节点

对每个候选 Ontology 节点：

### Ontology 候选: <能力域名>

**信号类型**：接口+多实现 / 独立包导出 / 注册机制 / 配置可替换
**源码证据**：
- <具体的 interface/ABC 定义位置:行号>
- <具体的多个实现位置，每个列出行号>
**下属 Entity**：<哪些上面提取的 Entity 属于这个能力域>
**判断置信度**：高（≥3 个实体信号）/ 中（2 个信号）/ 低（仅 1 个信号）

如果某个 Entity 找不到对应的 Ontology 节点，单独列出：

### 孤立 Entity（未找到 Ontology 归属）
- <Entity 名>：<原因>

## 输出

将所有产出写入：<输出路径>
不要在对话中写内容，必须写入文件。

## 核心约束

1. 每条事实声明必须有 ^[文件路径:行号] 结尾
2. Ontology 节点不能凭空发明——每个判定必须有源码中的结构信号支撑
3. 广泛探索，不要漏掉任何独立目录或独立包
4. 如果一个 Entity 同时匹配多个 Ontology 节点的信号，标注出来，不做强行归并
```

### 本轮已完成的提取（无需重跑）

- openclaw → `structural-entity-ontology-openclaw.md`（29 Entity, 24 Ontology）
- hermes → `structural-entity-ontology-hermes.md`（26 Entity, 10 Ontology）
- nanobot → `structural-entity-ontology-nanobot.md`（23 Entity, 12 Ontology）

三个 agent 独立运行，互不知晓。提取标准完全一致（相同的判断信号）。

## 三、Phase 2: 跨仓库 Ontology 对齐

### 提示词模板（通用）

```
你是跨仓库知识整合 agent。多个仓库已经各自独立完成了 Structural Entity 和候选 Ontology 的提取。

## 输入

<仓库 A>：<Phase 1 输出路径 A>
<仓库 B>：<Phase 1 输出路径 B>
<仓库 C>：<Phase 1 输出路径 C>（如有）

每个仓库的提取是完全独立的，互相不知道对方的存在。现在你需要对 Ontology 节点做跨仓库对齐。

## 任务：Ontology 节点对齐

### 步骤 1：识别共同 Ontology 节点

对每个仓库的候选 Ontology 节点，判断其他仓库中是否存在描述同一能力域的节点。

判断标准——以下条件**全部满足**即为同一能力域：
1. **核心抽象相同**：两边定义的核心 interface/ABC，暴露的方法签名和参数结构描述的是同一种操作
2. **职责范围一致**：解决的子系统问题相同（如"提供命令执行环境的可替换后端"）
3. **扩展机制同构**：多实现的结构信号是同一类型（都是接口+impl / 都是注册机制 / 都是配置切换）

注意：
- 名称不同不是否决条件（如 "Execution Environment Backend" vs "Terminal Execution Environments"）
- 实现数量不同不是否决条件（A 有 3 个 impl、B 有 9 个 impl 仍可对齐）
- 如果无法确定是否同一能力域，标注为"待人工确认"

### 步骤 2：输出对齐结果

对每个对齐后的 Ontology 节点：

### Aligned Ontology: <统一名称>

**涉及仓库**：<仓库 1, 仓库 2, ...>
**各仓库原始名称**：<A 叫 X, B 叫 Y>
**信号类型**：<接口+多实现 / 注册机制 / 配置可替换 / 独立包导出>
**对齐依据**：<为什么判断它们描述同一能力域——引用两边的 interface/ABC 定义>
**跨仓库实例**：
| 仓库 | Entity | 关键 interface/注册函数 | 实现数量 | 源码证据 |
|------|--------|----------------------|---------|---------|
| <A> | <Entity 名> | <interface 定义:行号> | <N 个> | <路径:行号> |
| <B> | <Entity 名> | <interface 定义:行号> | <M 个> | <路径:行号> |

### 步骤 3：列出独有 Ontology 节点

对每个仓库中无法与其他仓库对齐的 Ontology 节点，单独列出：
- <仓库名>：<Ontology 节点名> — 仅该仓库具备，待后续仓库覆盖

## 输出

写入：<Phase 2 输出路径>
必须在文件中输出，不在对话中写内容。

## 核心约束

1. 对齐判断必须基于两边的源码证据（interface 定义），不能仅凭名称相似
2. 不确定时标注为"待人工确认"，不强配
3. 不要因为实现数量不同而拒绝对齐
```

### 本轮执行

| 对齐轮次 | 输入 | 输出 |
|---------|------|------|
| R2 | openclaw + hermes Phase 1 产出 | `phase-2-alignment-r2.md` |
| R3 | R2 对齐结果 + nanobot Phase 1 产出 | `phase-2-alignment-r3.md` |

## 四、Phase 3a: 域内 Concept 提取（实验核心）

### 提示词模板（通用）

```
你是一个设计知识提取 agent。你的任务是对**一个已对齐的 Ontology 节点**下的跨仓库实例，提取 Concept 页。

## 输入

1. 当前 Ontology 节点的对齐信息（Phase 2 输出中该节点的条目）
2. 各仓库的 Phase 1 输出中，该 Ontology 节点下属的 Entity 详情（含关键机制、源码行号）

## 背景

以下仓库已经各自独立提取了 Structural Entity 和 Ontology 候选节点，然后由 Phase 2 完成了跨仓库 Ontology 对齐。你拿到的输入已经是"确认属于同一能力域"的跨仓库实例聚合。

你不需要判断"这些 Entity 是不是同一类"——Phase 2 已经完成了 Ontology 对齐。你只需要分析：在这个能力域下，不同仓库做了哪些**不同的设计选择**，以及为什么。

## 任务：提取 Concept（Design Knowledge Unit）

对每个 Ontology 节点下存在 ≥2 个仓库实例的，提取以下内容：

### 第一步：识别设计子问题

在这个能力域下，哪些设计决策是每个仓库都必须做的？这些不是实现细节，而是**架构级设计选择**——改变后会显著影响系统的扩展边界、调用约定或约束条件。

对每个设计子问题，用标准化问题陈述：
"如何在 [决策时机/场景] 决定 [决策对象] 的 [决策维度]？"

提取标准（必须满足）：
- 能写出 ≥2 个相互制约的**核心关切**——即满足关切 A 会增大满足关切 B 的成本
- ≥2 个仓库在该子问题上有**不同的权衡位置**——如果所有仓库答案相同，这是行业共识不是 Concept
- 二次开发者想替换或扩展这个能力时，这个选择会**限制或引导他的方案**

### 第二步：分析各仓库的权衡位置

对每个设计子问题：

| 仓库 | 权衡位置名称 | 具体实现 | 优先满足的关切 | 接受妥协的关切 | 源码证据 |
|------|------------|---------|--------------|-------------|---------|
| <A> | <一句话> | <关键机制描述> | <关切列表> | <关切列表> | ^[文件:行号] |

### 第三步：输出 Concept 页

```markdown
# <Ontology 节点名> — <设计子问题名>

**问题陈述**：如何在 [场景] 决定 [对象] 的 [维度]？

**核心关切**：
- 关切 1：...（一句话，说明一种必须被平衡的需求）
- 关切 2：...
（关切是解释"为什么没有唯一最优解"——满足一个会增大满足另一个的成本）

---

## 已知权衡位置

### 位置 A：<名称>
**优先满足**：关切 X、关切 Y
**接受妥协**：关切 Z
**特征**：
**关键机制**（源码可见）：^[文件:行号]
**代价**：
**已知实例**：<仓库名>

### 位置 B：<名称>
（同上，≥2 个不同位置）
```

## 溯源要求（强制）

- 每个仓库的"关键机制"必须包含 ≥1 个源码可见的方法名/配置值+行号
- 不信任 Phase 1 的简述——必须回溯 Phase 1 Entity 详情中列出的具体行号
- 如果 Phase 1 信息不足以支撑机制描述，标注并在末尾附"待验证"清单

## 输出

写入：<Concept 输出路径>
必须在文件中输出。

## 核心约束

1. 只在这个 Ontology 节点的范围内分析，不涉及其他能力域
2. 每个设计子问题必须能写出 ≥2 个相互制约的关切
3. 每个权衡位置必须有源码行号证据
4. 如果某个 Ontology 节点下所有仓库的实现方式相同，诚实标注"无有判別力的设计差异"
```

### R2 实验执行计划

Phase 2 对齐后，选出 ≥2 仓库共享的 Ontology 节点。对每个节点，按以下流程执行：

**3 次独立运行**：
- 用相同的 ontlogy 对齐结果和相同的 Entity 详情
- 3 个独立 subagent，独立上下文窗口
- 每个 subagent 独立产出 Concept 页
- 每次运行使用**完全相同的 prompt 模板 + 完全相同的输入材料**

```
输出路径命名：
  R2/<Ontology-slug>/concept-run-1.md
  R2/<Ontology-slug>/concept-run-2.md
  R2/<Ontology-slug>/concept-run-3.md
```

**Concept 子问题 inter-run 稳定性计算**：
- 从 3 次运行的产出中提取"识别到的设计子问题"集合
- 计算 Jaccard 系数：3 次 运行识别出的共同子问题数 / 并集子问题数
- > 0.7 视为稳定

### R3 实验执行计划

Round 3 加入 nanobot 后：
1. Phase 2 产出更新后的 Ontology 对齐（含 nanobot 实例）
2. 对已变更的 Ontology 节点（有新仓库加入），重新提取 Concept
3. 重复 3 次独立运行

**增量稳定性评估**：
- 对比 R2 和 R3 的 Concept 页
- R2 已有的分析是否被保留（追加不破坏）？
- R3 是否只增加了 nanobot 的行，还是修改了已有仓库的描述？

## 五、Phase 3b: Concept 独立验证

### 提示词模板（通用）

```
你是技术事实核查员。独立验证一个 Concept 页的每个技术 claim。

## 待验证的 Concept 页
<Concept 页路径>

## 验证方法

### A. 源码存在性验证
对每个仓库描述中出现的：
- 文件路径 → 确认文件存在
- 方法名/函数名 → 确认在对应文件中存在
- 行号 → 确认代码与描述一致
- 常量/配置值 → 确认实际值与描述一致

### B. 关切-实现对应验证
检查每条"核心关切"是否在跨仓库对比表中有对应体现。
一条关切如果在所有仓库的描述中都找不到对应的实现证据，标记为 ⚠️ 悬空关切。

### C. 权衡位置分类验证
检查每个仓库被分配的权衡位置是否准确：
- 该仓库的实现是否确实体现了该位置的"优先满足的关切"？
- 是否确实接受了该位置的"接受妥协的关切"所对应的代价？

### D. 绝对化语言验证
标记所有"所有"/"每次"/"永不"/"自动"/"完全"，验证是否有源码边界条件支撑。

## 输出格式

# 验证报告：<concept-slug>

## 格式完整性
- [ ] 问题陈述是"如何..."问题形式
- [ ] 核心关切列表 ≥2 条
- [ ] 每个权衡位置有"优先满足"和"接受妥协"两个字段
- [ ] 溯源表完整

## 逐仓库验证
### <仓库名>
- Claim: [引用]
  源码：^[文件:行号] [代码摘要]
  判定：✅/⚠️/❌ [修正建议]

## 关切验证
- 关切 1：✅ 在对比表中有体现 / ⚠️ 悬空
- 关切 N：...

## 汇总
总 claim 数：N | ✅：X | ⚠️：Y | ❌：Z

## 核心约束
1. 必须读源码，不能只看 Concept 页
2. 对 ⚠️ 和 ❌ 必须给出具体源码引用和修正建议
3. 特别关注绝对化语言和悬空关切
```

## 七、关键评估指标

| 指标 | 测量方式 | 预期 |
|------|---------|------|
| **Concept 设计子问题 inter-run 一致率** | 3 次独立运行产出设计子问题的 Jaccard | 实验组（有 Ontology 锚定）> 旧流程（无 Ontology，V2 baseline 参考值） |
| **源码准确性** | Phase 3b 验证通过率（✅/总数） | ≥ V2 实验的 83%（17 errors → 0 errors after fix） |
| **R2→R3 增量稳定性** | R3 追加 nanobot 后，R2 已有仓库描述是否被修改 | 已有分析不受影响，仅追加新行 |
| **Ontology inter-run 独立一致性** | 3 个仓库各自独立提取，事后对比 Ontology 节点重叠度 | 高置信节点重叠应 > 60% |
| **遗漏率** | Phase 3b 发现的遗漏设计子问题数 | 每个 Ontology 节点 ≤ 1 |

## 八、与旧流程（V2 2026-06-17）的关键差异

| 方面 | 旧流程 V2 | 新实验 |
|------|---------|--------|
| Entity 提取 | 从维度叙事页间接提取（混合结构+语义） | 直接从源码提取（结构信号驱动） |
| Ontology 层 | 不存在（隐式包含在 Concept 中） | 显式独立层，每个节点有信号类型和源码证据 |
| Concept 输入 | Entity 页（无边类型标注） | Ontology 节点 + 已聚合的跨仓库实例 |
| LLM 核心任务 | "轴定位" + "权衡分析" | "域内设计差异提取 + 权衡分析" |
| 稳定性验证 | 无 inter-run 测试 | 3 次独立运行，计算 Jaccard |
| 增量验证 | R2→R3 追加测试 | 复用，但增加了增量对 Concept 影响的观察 |
