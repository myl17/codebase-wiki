# 高风险操作审批的分级策略

## 问题陈述

AI Agent 在执行高风险操作（shell 命令、文件写入、网络请求等）时，必须有人类授权机制。核心问题是：系统应对所有危险操作一视同仁地要求人工审批，还是建立分级机制自动处理明显安全/危险的案例、仅将模糊案例升级给人类？这个问题对所有需要执行能力的 AI Agent 框架都是普遍的——过度审批会拖慢效率，审批不足会造成安全风险。

## 已知答案图谱

### 方案 A：全量人工审批
- 特征：所有高风险操作（exec 类工具）在进入 LLM 视野前就已区分，执行时统一走异步审批协议，阻塞等待人类 owner 决策后才执行
- 优势：安全边界清晰明确，不存在自动判断失误导致的风险；`「请求许可」而非「请求原谅」`的安全哲学
- 劣势：高频使用场景下用户审批疲劳严重；不适用于异步消息平台（用户可能不在线，agent 无限阻塞）

### 方案 B：智能分级审批
- 特征：三层递进审批——YOLO 层自动放行已知安全命令，Smart 层用辅助 LLM 评估风险，只有模糊案例升级到 Manual 层让人工决定
- 优势：大幅降低用户审批疲劳——大部分命令被自动处理；辅 LLM 成本远低于人工等待成本；适配消息平台异步场景
- 劣势：辅助 LLM 的判断可能出错（幻觉风险）；分级边界需要维护和调优；安全哲学的薄弱点——用户可能过度信任自动审批

## 跨仓库对比

| | OpenClaw | Hermes Agent |
|---|---|---|
| 选择的方案 | 方案 A：全量人工审批 | 方案 B：智能分级审批 |
| 具体实现 | ExecApprovalRequest 异步审批协议，阻塞 agent 执行等待 owner 决策；支持 host / gateway 双路径审批 | 三层架构：Layer 0 YOLO 自动放行 → Layer 1 Smart（aux LLM 评估 auto-approve/deny/escalate）→ Layer 2 Manual（tirith + 正则匹配 + 交互式审批） |
| 付出的代价 | 每次 exec 操作都需人类在线等待审批，高频使用场景下体验下降；gateway 场景下用户不在线时 agent 永久阻塞 | 需要维护辅助 LLM 的评估质量；正则模式列表和分级边界需要持续调优；用户可能因自动审批而降低安全意识 |

## 设计权衡

**选择全量人工审批（方案 A）更合理的场景**：
- 部署环境对安全有极端要求（如生产服务器运维），不允许任何自动判断失误
- 用户始终在线且愿意参与每次审批决策（如个人工具场景）
- 高风险操作频率较低，审批疲劳不是主要矛盾

**选择智能分级审批（方案 B）更合理的场景**：
- 消息平台异步场景，用户无法保证实时在线响应审批请求
- 高风险操作频率高，全量人工审批会严重拖慢工作流
- 可以接受辅助 LLM 的判断作为第一道防线（而非最终防线），允许少量误判通过后由其他机制兜底

**关键取舍本质**：方案 A 的安全边界是外置的（人类做出所有决策），方案 B 的安全边界是内置的（系统做出初步判断后人类复核）。前者的信任模型是「不信任任何自动判断」，后者的信任模型是「信任辅助 LLM 处理常规案例，人类处理边缘案例」。
