# 上下文窗口治理策略

## 问题陈述

AI Agent 在多轮 tool calling 中持续累积消息——system prompt、user message、assistant response、tool call、tool result 层层叠加。有限上下文窗口中需要管理：保留什么、丢弃什么、什么时候触发清理动作、清理到什么程度、是否让 LLM 感知到清理行为。核心问题是：上下文治理应由单一策略（一次性压缩到安全水位）还是多层策略（按严重程度逐步清理）来执行？这个选择直接影响 agent 在长对话中的任务连续性、token 成本曲线，以及 LLM 对上下文状态变化的感知方式。

## 已知答案图谱

### 方案 A：单一压缩 + 可恢复性优先
- 特征：在硬阈值触发时执行一次完整的 compaction——压缩的目标不是最大化压缩率，而是优先保留活跃任务状态、批处理进度、最后一次用户请求，确保 agent 在压缩后能无缝继续执行未完成任务
- 优势：行为简单可预测——只有一个触发条件和一次压缩操作；可恢复性优先确保长任务的连续性，减少 agent 因丢失上下文而重复工作
- 劣势：压缩是 big-bang 操作——触发时一次处理大量 token，可能造成明显的延迟；不区分不同程度的 token 压力——所有「超出阈值」的情况被同质化处理

### 方案 B：动态百分比 + 用户分级通知
- 特征：以当前模型 context window 的百分比定义多个触发点——75% 触发压缩、85% 和 95% 各通知用户一次（只通知用户，不注入 LLM）；压缩失败退避冷却 600 秒防重试风暴
- 优势：自动适配不同模型的 context window 大小；多级通知让用户在压缩发生前有心理预期；退避冷却防止压缩失败引发的重试风暴
- 劣势：百分比阈值需要跨模型验证；多级通知只面向用户，LLM 可能在不知情的情况下遇到上下文不足的问题

### 方案 C：四层透明治理
- 特征：在每次 LLM 调用前自动执行四层上下文治理——（1）Backfill：扫描孤立 tool_use 并插入合成错误结果；（2）Microcompact：10 轮次前的 tool result 替换为 `[tool result omitted from context]`；（3）Budget：单结果截断至 `max_tool_result_chars`，超限内容持久化到文件；（4）Snip：按 token 预算从尾部裁剪历史，边界对齐到 user-turn
- 优势：多层渐进治理——不是等到「接近限制」才大幅压缩，而是在每轮调用前逐步清理；LLM 透明——治理操作在发给 LLM 前完成，不占用 LLM 的推理 token 数；Microcompact 保留语义线索（知道有 tool 调用发生过）但释放了结果内容的 token 空间
- 劣势：四层治理逻辑复杂，调试时需要理解每层的触发条件和输出；Microcompact 丢失了旧 tool result 的完整内容——如果 10 轮前的结果对当前推理有关键意义，LLM 将无法获取；Snip 的边界对齐到 user-turn 可能仍需丢弃中间的 assistant 消息；治理不在 LLM 感知范围内——LLM 不知道哪些内容已被截断或省略

## 设计权衡

**选择单一压缩 + 可恢复性优先（方案 A）更合理的场景**：
- 部署环境和模型相对固定，context window 大小已知且稳定
- 任务通常是长流程、多步骤的，压缩后可恢复性比压缩率更重要
- 希望上下文治理行为尽可能简单可预测——一个触发条件、一次操作

**选择动态百分比 + 用户分级通知（方案 B）更合理的场景**：
- 部署环境使用多种不同 context window 的模型
- 用户需要感知压缩发生的时机（多级通知的价值）
- 压缩行为需要与模型能力自然匹配——大窗口模型自动享有更长的未压缩对话

**选择四层透明治理（方案 C）更合理的场景**：
- 希望避免 big-bang 式压缩的延迟峰值——用渐进式微清理代替一次性大压缩
- 治理动作对 LLM 透明是设计目标——LLM 看到的是已经治理过的干净上下文
- Tool calling 轮次可能很多（几十甚至上百轮），需要分层次处理不同严重程度的 token 压力
- 不希望治理逻辑占用 LLM 的推理 token 数

**关键取舍本质**：方案 A 将治理定义为一次性紧急措施——只在接近限制时才介入；方案 B 将治理的触发与模型能力绑定——让治理自动适配不同模型；方案 C 将治理定义为持续的低强度维护——每次调用前都做轻量清理，避免积累到需要大幅压缩的程度。方案 C 追求的是「治理成本平滑化」，代价是每轮都有治理开销和逻辑复杂度。
