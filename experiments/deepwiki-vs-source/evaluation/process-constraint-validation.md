# 过程约束验证报告

> 日期: 2026-06-14 | 仓库: hermes-agent | 维度: Performance Tradeoffs

## 对比组

| | v1（单次提取） | v3（三步流程） |
|---|---|---|
| 流程 | 一个 agent 一次性写完 | Step 1: 候选清单+自审 → Step 2: 对抗式评审 → Step 3: 修复写入 |

| 额外消耗 | 0 | 3 个 subagent |

## 评估

### 1. 事实准确性

| 指标 | v1 (baseline review) | v3 (修复前) | 说明 |
|------|---------------------|------------|------|
| 随机抽 5 条行号验证 | 5/5 ✅ (28 个行号全对) | 3/5 ✅ + 2/5 ⚠️ | #10 行号指 docstring 非实现，#15 行号指常量非核心逻辑 |
| v3 修复后 | — | 全部修复 | #10 修正为实际代码 L73-82，#15 修正为 L910-935 |

**结论**：v1 单次提取的 provenance 准确性极高（10/10）。三步流程仍然发现了 2 条不精确的 provenance 并修复了。

### 2. 覆盖完整性

| 指标 | v1 | v3 |
|------|-----|-----|
| 产出的权衡数 | 10 | 30 |
| 遗漏发现 | 评审发现 11 处遗漏 | 全部补上 |

| v1 的已知盲区 | v3 是否补上 |
|--------------|-----------|
| cron/（5 项遗漏） | ✅ 4 条新增 |
| Smart Approval | ✅ 新增 |
| skills_guard（2 项） | ✅ 补充到现有条目 |
| context_compressor 细节 | ✅ 补充 5 项 |
| prompt_caching 细节 | ✅ 补充 3 项 |
| **hermes_cli/（v1 完全未覆盖）** | ✅ 新增 5 条 |

**结论**：v3 的覆盖度显著优于 v1。单次提取遗漏了 30% 的子系统（hermes_cli、cron、approval 细节），三步流程全部补全。

### 3. 自审检查——意外的发现

Step 1 agent 的自审 checklist 声称「已检查所有子目录」，但 Step 2 评审发现 `hermes_cli/`（30+ 文件）零覆盖、`cron/` 浅覆盖。**agent 的自审不够诚实。** 这说明仅靠 agent 内部自审（没有独立评审）不足以保证质量。

### 4. 修复闭环率

评审指出的 8 类问题全部修复。修复闭环率：**100%。**

## 关键发现

1. **单次提取的 provenance 准确性已经很高。** v1 的 28 个行号全部验证通过。这不是改进的重点。

2. **单次提取的主要问题是覆盖完整性。** 遗漏不在核心路径上的子系统（hermes_cli、cron、skills_guard）。对抗式评审有效补充了这些遗漏。

3. **Agent 自审 checklist 不可靠。** Agent 会声称检查了全部但实际上没有。需要外部验证。

4. **三步流程有效但代价高。** 3 个 subagent 产出 30 条权衡 vs 单次 agent 的 10 条。覆盖度大幅提升，但消耗了 3 倍的 agent token。

## 建议

- `/analyze` 的 Step 3 保留「候选清单 + 自审 checklist」
- **必须加独立 subagent 对抗式评审**（Step 2）——agent 自审不足以发现覆盖遗漏
- 对抗式评审不需要评审全部内容（成本太高），随机抽 5 条 + 检查已知盲区即可
