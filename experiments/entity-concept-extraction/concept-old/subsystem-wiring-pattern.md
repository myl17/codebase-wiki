# 子系统组装与连线方式

## 问题陈述

AI Agent 框架需要将 provider、bus、tools、memory、session、cron、skills 等众多子系统实例化并连接成一个可运行的 agent。核心问题是：这些子系统应该如何被组装——通过依赖注入容器将连线逻辑分散到各模块，还是通过中央编排器在一个构造入口集中完成所有连线？这个选择直接影响系统的可发现性（「这个系统有什么」能从哪里得到答案）、子系统间的耦合程度，以及单元测试的可行性。

## 已知答案图谱

### 方案 A：Plugin API 依赖注入
- 特征：框架提供 Plugin API（如 `OpenClawPluginApi` 的 25 个注册方法），每个模块在自己的 plugin 入口中独立调用注册方法注册自己的子系统（`registerContextEngine`、`registerMemoryCapability` 等）；框架内部通过 DI 容器在运行时按需组装
- 优势：各子系统独立注册，模块间耦合极低——新增一个子系统不需要修改任何已有文件；plugin 接口本身就是契约，IDE 类型检查对注册参数有效；各模块可以独立单元测试（mock plugin API 即可）
- 劣势：系统的完整结构需要阅读所有 plugin 的注册调用才能拼凑出来——「这个系统有什么」没有一个单一入口可以回答；25 个注册方法增加了新开发者的学习成本；DI 容器的组装逻辑本身是额外的复杂度

### 方案 B：中央编排器
- 特征：一个单一的中央类（如 `AIAgent`）在内部管理所有子系统的创建和引用——外部不直接接触子系统，只与中央编排器交互
- 优势：系统结构由中央编排器定义，外部视角简洁——只有一个入口类；子系统之间的交互可以在中央编排器中协调，不需要跨模块通信协议
- 劣势：中央编排器本身承载所有子系统的耦合——任何子系统变更都可能需要修改中央编排器；子系统无法独立单元测试（因为没有注入点，必须构造完整的中央编排器）

### 方案 C：单体 Hub 构造函数
- 特征：所有子系统的实例化和连线在一个构造函数中集中完成——读一个文件的一个函数即可得到「这个系统有什么」的完整答案；不引入 DI 容器或 factory pattern 的抽象层
- 优势：极致的可发现性——一个构造函数的代码流就是系统的装配说明书；不需要理解 DI 容器的工作方式或注册顺序约定；调试时不需要追踪注入链条
- 劣势：构造函数成为单一故障集中点——任何子系统的初始化失败都会导致整个 agent 不可用；子系统无法独立单元测试，因为没有接口注入点；所有子系统必须与 Hub 在同一进程中运行

## 跨仓库对比

| | OpenClaw | Hermes Agent | nanobot |
|---|---|---|---|
| 选择的方案 | 方案 A：Plugin API 依赖注入 | 方案 B：中央编排器 | 方案 C：单体 Hub 构造函数 |
| 具体实现 | `OpenClawPluginApi` 提供 25 个注册方法，每个 plugin 在入口文件中命令式调用注册自己的子系统；框架通过 DI 容器在运行时按需组装所有注册的子系统 | `AIAgent` 单一类管理完整的 tool-calling 循环，内部持有所有子系统的引用；外部通过 AIAgent 的接口间接与子系统交互 | `AgentLoop.__init__()` 在一个构造函数中实例化并注入 provider、bus、context、tools、runner、subagents、consolidator、dream、sessions、cron、heartbeat、MCP 全部子系统 |

## 设计权衡

**选择 Plugin API 依赖注入（方案 A）更合理的场景**：
- 需要 Plugin/扩展生态，第三方开发者需要独立注册子系统而不修改核心代码
- 子系统数量多且各自有复杂的生命周期管理需求
- IDE 类型检查和编译时验证对正确性很重要（TypeScript 优势）
- 「系统有什么」的完整图景通过文档或工具生成即可，不依赖读代码发现

**选择中央编排器（方案 B）更合理的场景**：
- 子系统数量中等但交互复杂，需要中央协调
- 外部 API 的简洁性比内部的模块独立性更重要
- 子系统的完整集合相对固定，不需要频繁扩展

**选择单体 Hub 构造函数（方案 C）更合理的场景**：
- 追求极致的代码可发现性——「读一个函数就能理解整个系统的装配」
- 子系统数量有限且相对稳定，构造函数的长度完全可控
- 不需要第三方扩展 agent 的核心子系统组装方式
- 单体部署模型下，所有子系统天然在同一进程中

**关键取舍本质**：方案 A 用系统的结构分散化换取模块的独立性和可扩展性——每个模块知道自己的职责，但没有人知道全貌；方案 B 用中央编排器换取外部 API 的简洁——全貌在中央，但子系统失去独立性；方案 C 用极致的集中化换取极致的可发现性——全貌在一个构造函数中，但为此牺牲了模块独立性和单元测试的可能性。
