# 技能来源与生命周期

## 问题陈述

AI Agent 的能力可以通过「技能」（Skills）来扩展——技能定义 agent 如何完成特定类型的任务。核心问题是：这些技能由谁创建和维护——人类编写静态的技能文件，还是 Agent 在完成任务的过程中自主创建和更新技能？这个选择直接决定了技能的演化速度、质量控制机制，以及系统对未知任务的适应能力。

## 已知答案图谱

### 方案 A：人类编写，静态文件
- 特征：技能由人类以 Markdown 文件形式预先编写，内容是 prompt 工程（任务说明、工作流步骤、命令快捷键等），通过 hook 机制在 prompt 组装阶段注入 system prompt；agent 只能使用技能，不能创建或修改技能
- 优势：技能质量完全由人类控制，不会出现 agent 自动生成的低质量或错误技能；技能内容可审计、可版本管理、可 review；安全性最高——不会出现 agent 被诱导创建恶意技能；适合作为零代码扩展层降低普通用户的定制门槛
- 劣势：技能库的扩展完全依赖人类投入，无法随使用自动增长；技能过时（工具版本升级、API 变更）后需要人类手动更新；无法从 agent 的成功经验中自动提取和复用模式

### 方案 B：Agent 自主创建与维护
- 特征：通过 system prompt 中的驱动指令（如 SKILLS_GUIDANCE）让 LLM 在完成复杂任务后主动调用 `skill_manage` 工具创建技能；技能以 SKILL.md 文件存储并在每次会话启动时全量注入 system prompt；agent 在使用中发现技能过时时可通过 patch 操作就地修复
- 优势：技能库随使用自动增长——每次成功完成复杂任务都可能产生新技能，越用越能干；agent 在使用中自主发现并修复过时技能，不需要人类维护；学习行为内嵌到 agent 决策流中，时机准确（刚完成复杂任务时立即提取经验）
- 劣势：技能质量依赖 LLM 的判断力，可能生成不准确或过度特化的技能；缺乏人类 review 环节，错误技能可能在被发现前反复使用；安全性较弱——恶意用户可能诱导 agent 创建危险技能

## 跨仓库对比

| | OpenClaw | Hermes Agent |
|---|---|---|
| 选择的方案 | 方案 A：人类编写，静态文件 | 方案 B：Agent 自主创建与维护 |
| 具体实现 | 纯 Markdown Skills 文件通过 `buildWorkspaceSkillsPrompt` + `buildWorkspaceSkillCommandSpecs` 注入 system prompt，作为最轻量的第三层扩展（零代码）；技能内容完全由人类维护，agent 不具备创建或修改技能的能力 | 三段 system prompt 驱动指令中的 SKILLS_GUIDANCE（完成 5+ tool calls 的复杂任务后主动创建技能）+ `skill_manage` 工具（create/edit/patch/delete）+ MEMORY_GUIDANCE 保存用户偏好 + SESSION_SEARCH_GUIDANCE 搜索历史；三层时间尺度（实时/跨会话/代际）的学习闭环 |
| 付出的代价 | 技能库增长完全依赖人类投入；过时技能需要人类手动发现和更新 | 技能质量无人类 review 兜底；LLM 可能创建低质量或冗余技能；安全风险——恶意指令可能诱导创建危险技能 |

## 设计权衡

**选择人类编写静态文件（方案 A）更合理的场景**：
- 目标用户是普通用户，通过编写 Markdown 即可定制 agent 行为，不需要理解 agent 自学习的概念
- 技能内容需要严格质量控制和版本管理
- 安全是第一优先级，不允许 agent 自动创建可执行的行为定义
- 技能的复杂度上限由 Markdown/prompt 工程表达力决定，不需要 agent 动态生成

**选择 Agent 自主创建与维护（方案 B）更合理的场景**：
- 用户无法或不愿意手动维护技能库，希望系统越用越聪明
- 任务类型多样且频繁变化，静态技能库难以预覆盖
- 可以接受一定比例的技能质量问题，通过使用中的 patch 机制自我修正
- 信任 LLM 对「什么值得固化为技能」的判断力

**关键取舍本质**：方案 A 的技能是「工具」——人类锻造、质量可控、稳定可靠；方案 B 的技能是「经验」——agent 在使用中积累、自动更新、持续演化。方案 A 的信任模型是人类作为唯一的知识权威，方案 B 的信任模型是 agent 作为知识的共同生产者。
