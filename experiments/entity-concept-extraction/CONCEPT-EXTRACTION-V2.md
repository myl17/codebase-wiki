# Concept 提取流程 V2

> 基于三轮增量实验（2026-06-17）的评估结论和后续分析讨论修订
> 上一版本问题：Concept 定义操作化不足、交叉审查依赖语义感知、多仓库视角缺失、格式无强制约束

---

## 核心概念重新定义

### 什么是 Concept

**Concept 是一个所有同类系统都必须回答的设计问题，这个问题没有唯一最优解，因为它涉及多个相互制约的关切（concerns），不同系统对这些关切的优先级排序不同，因此落在了不同的权衡位置上。**

几个推论：

1. **Concept 不是两个仓库的对比**，是一个所有同类系统都会面对的设计轴。轴上可以有 2 个、3 个、4 个已知位置，随仓库增量填充。
2. **没有唯一最优解的原因** 是多个关切相互制约——满足关切 A 会增大满足关切 B 的成本。关切数量没有上限，最少两个，通常 3-5 个。
3. **新仓库进来时**，不是问"它和谁不同"，而是"在这些关切下，它把权衡点放在了哪里，为什么"。
4. **Concept 名称是问题**，不是答案。命名格式：`{能力域}-{决策维度}`，例如 `skill-injection-granularity`、`memory-retrieval-timing`。

### 什么不是 Concept

- **实现细节**：换一种实现方式，系统的约束和行为不变（例如"用 SQLite 还是 PostgreSQL"）
- **技术选型**：选择影响性能和运维成本，但不影响其他系统对它的调用方式（例如"用哪个 LLM provider"）
- **本仓库独有**：这个设计问题只有一个仓库面对，其他同类系统根本不需要决策（进种子库待观察，不升级）

**三层过滤测试**（在 Phase 1 中使用）：

```
层 1 — 实现细节（不进种子库）
  判断：换一种实现，系统对外的约束和调用方式不变
  示例：日志用 JSON 还是纯文本格式

层 2 — 技术选型（可进种子库，不升级 Concept）
  判断：影响性能/成本，但接口契约不变，调用方无感知
  示例：embedding 模型的选择

层 3 — 架构决策（进种子库，候选 Concept）
  判断：选择改变了系统的扩展边界、调用约定或约束条件，
        二次开发者必须了解才能正确扩展或替换这个能力
  示例：记忆后端是独占替换还是加性叠加——影响扩展接口设计
```

### Concept 的判断视角

提取设计选择时，用两个视角过滤：

**架构师视角**：如果一个架构师在设计新的同类框架时面对同样的问题，他会遇到相同的取舍吗？
- 过滤掉：具体库的选择、内部优化手法、单仓库特有的历史包袱

**二次开发者视角**：如果一个开发者想替换或扩展这个能力，这个选择会限制或引导他的方案吗？
- 过滤掉：不对外暴露的内部实现、不影响扩展点的内部机制

**满足任意一个视角即可进入候选。**

## 流程总览

```
每个新仓库进来时执行以下流程：

Phase 1：设计选择提取
  输入：新仓库的 Entity 页 + 维度叙事页
  输出：新仓库设计选择草稿（格式与种子库一致）
  关键：独立提取，不读已有种子库

Phase 2：轴定位（替代"交叉审查"）
  输入：Phase 1 草稿 + 已有种子库 + 已有 Concept 页
  输出：候选 Concept 清单（新轴 / 已有轴新位置 / 追加同位置）
  关键：不做语义配对，做轴定位

Phase 3a：Concept 写作（per-Concept 独立 agent）
  输入：候选 Concept 清单中的单个条目 + 溯源路径
  输出：Concept 页（新建 / 追加更新）
  关键：强制读源码验证，格式严格

Phase 3b：独立验证（per-Concept 独立 agent）
  输入：Phase 3a 写的 Concept 页
  输出：验证报告（✅ / ⚠️ / ❌）

Phase 3c：修复
  输入：验证报告 + Concept 页
  输出：修复后的 Concept 页

Phase 3d：种子库更新
  输入：新仓库草稿 + Phase 3a-3c 产物
  输出：更新后的种子库
```

## Phase 1 提示词：设计选择提取

```
你是一个代码仓库知识提取专家。从 <仓库名> 的 Entity 页中提取所有架构级设计选择，
生成设计选择草稿。

**不要读已有种子库或其他仓库的任何结果。独立提取。**

## 输入
- Entity 页：<路径>/*.md
- 维度叙事页（按需验证细节）：<路径>/dimensions/

## 什么值得提取

每个候选设计选择必须通过三层过滤：

层 1（不提取）：实现细节
  判断：换一种实现，系统对外的约束和调用方式不变
  示例：用哪种日志格式、用哪个第三方库

层 2（可提取）：技术选型
  判断：影响性能/成本，但不影响调用约定
  说明：进草稿，但不会升级为 Concept

层 3（优先提取）：架构决策
  判断：改变了系统的扩展边界、调用约定或约束，
        二次开发者必须了解才能正确扩展此能力

过滤时用两个视角检验（满足任意一个即可）：

  架构师视角：设计新的同类框架时，这个取舍是否普遍存在？
  二次开发者视角：想扩展或替换这个能力时，这个选择是否会限制或引导方案？

## 输出格式

路径：<草稿路径>.md

每条设计选择用以下格式：

---

## <设计问题名>（如何...形式）

**维度**：<Architecture / Extension Points / Performance Tradeoffs / Dependency Strategy / Testing Philosophy>
**问题陈述**：如何...（一句话，问题形式，说明这类系统为什么必须面对这个问题）
**核心关切**：
- 关切 1：...（一句话，说明一种需要被满足但和其他关切存在制约的需求）
- 关切 2：...
- 关切 N：...（列举所有相互制约的关切，不预设数量）

| 仓库 | 选择 | 简述 | 溯源 |
|------|------|------|------|
| <仓库名> | <方案名（名词短语）> | <一句话> | <Entity页路径 → 源码文件路径> |

**层级**：层 2 技术选型 / 层 3 架构决策

---

## 核心约束

1. 独立提取，不读任何其他仓库的内容
2. 每个条目必须填写"核心关切"——写不出来说明这不是真正的设计取舍
3. 溯源必须包含 Entity 页路径 + 源码文件路径（从 Entity 页"关键实现"节提取）
4. 问题陈述是「如何...」的问题形式，不是答案描述
5. 只提取有明确替代方案的选择——"用了 TypeScript"不是选择，"工具权限在消息处理路径上同步门控而非事后审计"才是
```

## Phase 2 提示词：轴定位

```
你是一个跨仓库知识整合专家。新仓库（<仓库名>）的设计选择草稿已完成。
现在将草稿中的每个条目在已有种子库中定位，识别候选 Concept。

## 输入

1. 新仓库设计选择草稿：<草稿路径>
2. 已有种子库：<种子库路径>（包含已有仓库的全部设计选择，部分已升级为 Concept）
3. 已有 Concept 页（需要时参考）：<concept目录>
4. 新仓库 Entity 页（判断不确定时按需回溯）

## 核心操作：轴定位

对草稿中的每条设计选择，执行以下四步判断：

### 步骤 1：找到对应轴

扫描已有种子库，找维度相同且能力域相同的条目。

"能力域相同"的判断标准：
  两个条目的问题陈述如果做了回答，它们描述的是系统的同一个能力或子系统。
  示例：「如何让技能内容在 LLM 调用时可见？」和「如何在 token 预算内管理技能注入量？」
  → 都是关于技能注入能力的设计选择，能力域相同

### 步骤 2：标准化问题陈述

如果找到候选匹配轴，将两边的问题陈述用统一模板重写：

  如何在 [决策时机/场景] 决定 [决策对象] 的 [决策维度]？
  其中 [决策维度] 反映了核心关切列表中的主要张力

如果重写后两条问题陈述的 [决策对象] 和 [决策维度] 相同，则认为是同一轴。

### 步骤 3：在轴上定位

确认是同一轴之后，判断新仓库的选择在该轴上的位置：

  类型 A（新位置）：新仓库的选择和已有所有选择的权衡侧重都不同
    → 标记为"候选 Concept 更新：新位置"

  类型 B（已有位置）：新仓库的选择和已有某个选择的权衡侧重一致
    → 标记为"追加同位置实例"，注明与哪个仓库一致以及实现差异

  类型 C（已有轴，Concept 还未升级）：
    当前条目的 Concept 状态是"待观察"，新仓库带来了第二个不同位置
    → 标记为"候选 Concept 升级：首次出现第二位置"

### 步骤 4：开辟新轴或标记独有

如果扫描后没有找到对应轴：

  a. 回溯已有仓库对应子系统的 Entity 页，确认已有仓库是否也面对这个问题
     - 面对但漏提 → 补充已有仓库的条目，按类型 A/B/C 处理
     - 确实不面对 → 标记"新轴：仅<仓库名>—待观察"

  b. 用核心关切验证新轴的合法性：
     能写出 ≥2 个相互制约的关切 → 合法新轴
     写不出来 → 可能是实现细节，不进种子库

## 反向检查

对已有种子库中所有"仅<已有仓库>—待观察"的条目逐条检查：
  新仓库是否也面对这个问题？
  → 是：补充新仓库条目，按类型 A/B/C 处理
  → 否：保持状态不变

## 输出格式

### 候选 Concept 清单

每条用以下格式输出：

---

候选 Concept #N：<slug>（{能力域}-{决策维度} 格式）

**操作类型**：新轴升级 / 已有轴新位置 / 已有轴首次第二位置 / 追加同位置实例

**标准化问题陈述**：如何在 [决策时机/场景] 决定 [决策对象] 的 [决策维度]？

**核心关切**（综合两仓库的草稿关切列表，合并去重）：
- 关切 1：...
- 关切 2：...
- 关切 N：...

**涉及的种子库条目**：#编号（已有仓库）+ 草稿条目名（新仓库）

**各仓库权衡位置**：
| 仓库 | 权衡位置名称 | 优先满足的关切 | 接受妥协的关切 |
|------|------------|--------------|--------------|
| <仓库A> | | | |
| <仓库B> | | | |
| <仓库N> | | | |

**溯源**：
| 仓库 | 必须读的源码文件 |
|------|----------------|
| <仓库A> | |
| <仓库B> | |

---

## 核心约束

1. 操作是"在轴上定位"，不是"语义相似度配对"
2. 轴的合法性由核心关切验证——写不出相互制约的关切，不是真正的设计选择
3. 判断"能力域相同"时按需回溯 Entity 页，不依赖问题陈述的字面相似度
4. 候选 Concept 清单必须包含溯源路径——这是 Phase 3a 必须读的源码文件
5. 一个仓库独有的设计选择是正常的，不要强行配对
```

## 追加更新的额外约束

如果操作类型是"追加新位置"或"追加同位置实例"，必须先声明追加语义：

**追加语义声明**（在文件修改前输出）：
- 追加类型：新位置（新增权衡位置小节） / 同位置实例（在已有位置的"已知实例"追加）
- 修改范围：将新增/修改的具体小节列出
- 保留承诺：已有仓库的描述不做实质性修改（除非源码验证发现错误）

**禁止**：未声明追加语义就直接追加——这是导致"前半部分仍然只写两个仓库"的直接原因。

## 核心约束

1. 必须读源码，不信任种子库和 Entity 页的简述
2. 每个仓库的"具体实现"必须包含 ≥1 个源码可见的机制细节（方法名+行号）
3. 核心关切列表中每一条都要能对应到跨仓库对比表的某一行——如果关切在对比表里找不到体现，删掉那条关切
4. 追加更新时必须先输出追加语义声明
5. Concept 名称是问题（如何...），不是答案，不是方案名

## Phase 3b 提示词：独立验证

```
你是一个技术事实核查员。独立验证一个 Concept 页的每个技术 claim。

## 待验证的 Concept 页
[路径]

## 验证方法

### A. 源码存在性验证
对每个仓库描述中出现的：
- 文件路径 → 确认文件存在
- 方法名/函数名 → 确认在对应文件中存在
- 行号 → 确认代码与描述一致
- 常量/配置值 → 确认实际值与描述一致

### B. 关切-实现对应验证
检查每条"核心关切"是否在跨仓库对比表中有对应体现。
一条关切如果在所有仓库的描述中都找不到对应的实现证据，标记为 ⚠️ 悬空关切。

### C. 权衡位置分类验证
检查每个仓库被分配的权衡位置是否准确：
- 该仓库的实现是否确实体现了该位置的"优先满足的关切"？
- 是否确实接受了该位置的"接受妥协的关切"所对应的代价？

### D. 绝对化语言验证
标记所有"所有"/"每次"/"永不"/"自动"/"完全"，验证是否有源码边界条件支撑。

### E. 追加完整性验证（如果 Concept 页有多个仓库）
检查所有提到已有仓库的叙述节（问题陈述、已知权衡位置小节、选择指南）是否已
更新为反映全部已有仓库，而不是停留在早期版本（只提两个仓库）。

## Phase 3d 提示词：种子库更新

```

## 与 V1 流程的关键变化对比

| 方面 | V1 | V2 |
|------|----|----|
| Concept 定义 | 同一问题的不同答案（两仓库配对思维） | 设计轴上的多个权衡位置（多仓库增量思维） |
| Phase 2 操作 | 语义配对（"同一句如何..."） | 轴定位（四步：找轴→标准化→定位→验证合法性） |
| 无唯一最优解的原因 | 隐含两端张力 | 显式列出 N 个相互制约的关切（≥2，不预设上限） |
| 提取过滤标准 | "有替代方案的选择" | 三层测试（实现细节/技术选型/架构决策）+ 两个视角 |
| Concept 页格式 | 方案 A/B/C + 设计权衡 | 权衡位置 A/B/N + 核心关切 + 优先/妥协的关切 |
| 追加更新 | 无明确规约 | 必须先声明追加语义（新位置/同位置实例/反例） |
| 验证覆盖 | 源码事实核查 | 源码核查 + 关切-实现对应 + 追加完整性 |
| 命名规范 | 问题形式（无结构） | {能力域}-{决策维度} 格式 |
