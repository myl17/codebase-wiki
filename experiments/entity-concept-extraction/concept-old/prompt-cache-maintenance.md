# Prompt Cache 维护策略

## 问题陈述

LLM API 的 Prompt Caching 机制允许稳定前缀部分被缓存复用，大幅降低多轮对话的 token 输入成本。但缓存的命中依赖于前缀内容在多轮之间保持稳定。AI Agent 框架的核心问题是：应通过精心设计 prompt 内容结构来人为制造缓存的稳定/动态边界，还是通过复用运行时会话实例来自然保持前缀的连续性？这两种策略代表了对同一个成本优化问题的不同工程哲学。

## 已知答案图谱

### 方案 A：内容锚定
- 特征：在 system prompt 中显式插入缓存边界标记（如 `CACHE_BOUNDARY`），将 prompt 切分为稳定前缀（技能定义、人格描述、工具规范等）和动态后缀（当前任务上下文、记忆注入、历史对话等）；稳定部分打上 `cache_control: ephemeral` 标记，动态部分不影响缓存命中
- 优势：缓存策略由内容控制，不依赖会话管理；即使 agent 实例重建，只要稳定前缀相同缓存就能命中；精细控制缓存粒度
- 劣势：system prompt 结构复杂度上升，需要维护缓存边界标记的正确位置；缓存效果依赖于 prompt 结构的稳定性——任何对稳定前缀的修改都会导致缓存失效

### 方案 B：会话复用
- 特征：在 Gateway 场景下跨消息缓存 AIAgent 实例，不销毁重建，以维持 LLM API 连接的 prompt cache prefix 在多轮消息之间连续有效；每条新消息复用已有 agent 实例而非新建
- 优势：实现简单——不需要修改 prompt 内容结构，仅通过控制实例生命周期即可获得缓存收益；与 LLM provider 的缓存实现无关，协议层透明
- 劣势：缓存收益局限于同一 agent 实例的生命周期；实例长期存活会占用内存；缓存命中依赖会话连续性——用户切换会话或 agent 实例被回收后缓存失效；不适用于无状态部署（serverless、水平扩展）

### 方案 C：声明顺序稳定化
- 特征：不修改 prompt 内容结构，也不复用运行时会话——通过强制工具定义的排序保持 system prompt 前缀在多次调用间稳定：builtins 按名称排序在前，MCP 工具（`mcp_` 前缀）在后；只要不新增/删除 builtin 工具，system prompt 前缀在整个会话生命周期中不变
- 优势：实现极简——仅需要在 `get_definitions()` 中对工具列表排序，不涉及内容锚定标记也不依赖实例缓存；与 Anthropic prompt cache 的自动前缀匹配机制天然对齐——稳定的工具定义顺序 = 稳定的缓存前缀；兼顾了无状态部署兼容性（不需要实例缓存）和低工程复杂度（不需要内容锚定边界标记）
- 劣势：缓存收益的稳定范围仅覆盖工具定义部分——system prompt 中其他动态内容（如记忆、技能摘要）不在稳定化范围内；新增/删除内置工具会使所有活跃会话的缓存失效；仅适用于 prompt cache 自动识别前缀的 provider（如 Anthropic），对不支持自动缓存或缓存策略不同的 provider 无效果

## 设计权衡

**选择内容锚定（方案 A）更合理的场景**：
- 系统需要在无状态环境下部署（serverless、容器化水平扩展），agent 实例不能跨请求存活
- 有明确的 prompt 内容边界——稳定部分和动态部分可以清晰切分
- 缓存命中希望跨会话、跨实例生效——只要 prompt 结构稳定，新实例也能命中缓存

**选择会话复用（方案 B）更合理的场景**：
- 系统部署在长生命周期进程中（如 Gateway 常驻进程），实例缓存是自然的副作用
- Gateway 场景下同一条消息链路的多次交互共享同一 agent 实例是直观的
- 不想在 prompt 内容层面做缓存相关的工程改造——通过运行时优化即可获得缓存收益

**关键取舍本质**：方案 A 追求缓存的**可移植性**——缓存策略编码在内容中，不绑定任何运行时状态；方案 B 追求缓存的**简单性**——利用已有的实例生命周期管理自然获得缓存收益，不做额外的内容工程。两者可以互补使用：内容锚定保证跨实例的缓存命中，会话复用降低同一实例内部的重复 token 消耗。但实际上，当前两个仓库各自只实现了一种策略。
