> 使用记录：2026-06-20 hermes-agent

```
你的任务是直接从 <仓库名> 源码中提取 Structural Entity 和候选 Ontology 节点。

## 输入源

<仓库名> 源码在 <源码路径>。
这是你的唯一信息来源。不用训练数据中可能存在的知识。

## 工作方式

你自己决定读哪些文件。广泛探索直到全面理解这个仓库的结构。

## 第一步：提取 Structural Entity

对每个 Entity，你需要产出的是一个**有技术深度的页面**——不是只标注文件路径。格式：

### Entity: <名称>

**代码位置**：<目录/包路径>
**这个模块解决什么问题**：<一句话>
**对外暴露什么**：<关键类/函数/接口，含文件路径:行号>
**它和谁交互**：<依赖的其他模块/被谁调用>
**为什么它是可分离的**：<独立目录？独立接口？独立的 npm 包/package？>

**关键机制**（源码可见）：
- <机制 1>：<描述> ^[文件路径:行号]
- <机制 2>：...

**源码证据**：
- 入口文件：<路径>
- 核心类型/接口定义：<路径:行号>

## 第二步：提取候选 Ontology 节点

Ontology 节点 = 跨仓库可对齐的"能力域"。但你不要凭空创造分组——**你只标注原始开发者已经通过代码结构表达出来的分组**。

判断标准（**必须基于源码证据，不能凭空判断**）：
1. **接口/抽象类 + 多个实现**：源码中是否存在一个 interface/ABC，后面跟着 ≥2 个不同的 impl？→ 这个接口定义的能力域 = 候选 Ontology 节点
2. **独立的包/目录 + 对外的类型/函数导出**：是否存在独立的目录，有明确的入口文件，导出类型/接口？→ 这个包定义的能力域 = 候选 Ontology 节点
3. **注册机制**：是否存在 register*()、add_*() 等显式的注册函数，允许外部添加新实现？→ 注册的对象类型所代表的能力域 = 候选 Ontology 节点
4. **配置项中的可替换策略**：是否存在配置项允许用户在多个策略中切换？→ 这个配置项控制的能力域 = 候选 Ontology 节点

对每个候选 Ontology 节点：

### Ontology 候选: <能力域名>

**信号类型**：接口+多实现 / 独立包导出 / 注册机制 / 配置可替换
**源码证据**：
- <具体的 interface/ABC 定义位置:行号>
- <具体的多个实现位置，每个列出行号>
**下属 Entity**：<哪些上面提取的 Entity 属于这个能力域>
**判断置信度**：高（≥3 个实体信号）/ 中（2 个信号）/ 低（仅 1 个信号）

如果某个 Entity 找不到对应的 Ontology 节点（不属于任何已有的可分离能力域），单独列出：

### 孤立 Entity（未找到 Ontology 归属）
- <Entity 名>：<原因——没有可替换策略/没有多个实现/不属于接口驱动>

## 输出

将所有产出写入：<输出路径>

不要在对话中写内容，必须写入文件。

## 核心约束

1. 每条事实声明必须有 ^[文件路径:行号] 结尾
2. Ontology 节点不能凭空发明——每个判定必须有源码中的结构信号支撑
3. 广泛探索，不要漏掉任何独立目录或独立包
4. 如果一个 Entity 同时匹配多个 Ontology 节点的信号，标注出来，不做强行归并
```
