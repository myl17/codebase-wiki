# 上下文压缩的触发机制

## 问题陈述

AI Agent 在长对话中需要压缩上下文以避免超出 LLM 的 context window 限制。核心问题是：压缩何时触发——按绝对 token 数量设置固定阈值，还是按当前模型 context window 的百分比动态计算？这个选择直接影响不同模型上的压缩行为一致性、用户对压缩时机的可预期性，以及压缩策略对模型升级的适应能力。

## 已知答案图谱

### 方案 A：硬 token 阈值
- 特征：设置固定的 token 数量阈值（如 16000 hard / 32000 soft），不随模型 context window 大小变化；达到阈值即触发压缩或警告，行为在所有模型上完全一致
- 优势：行为完全可预测——无论使用哪个模型，压缩都在相同的 token 消耗水平触发；调试简单，阈值是固定常数；适合 context window 差异不大的模型族
- 劣势：切换到大 context window 模型时，固定阈值可能过早触发压缩，浪费可用空间；切换到小 context window 模型时，固定阈值可能触发过晚甚至无法触发就已经超限；阈值需要根据部署的主流模型手动调整

### 方案 B：动态窗口百分比
- 特征：以当前模型 context window 的百分比定义触发阈值（如 75% 触发压缩、85% 和 95% 各通知用户一次）；不同模型自动适配不同的绝对 token 数
- 优势：自动适配不同 context window 大小的模型，无需手动调整阈值；百分比语义清晰——「用了四分之三窗口时压缩」在任何模型上都合理；模型升级（context window 增大）时自动获得更多可用空间
- 劣势：行为随模型变化——切换模型后压缩时机不同，可能导致用户体验不一致；百分比阈值的选择需要跨模型验证，过大或过小的百分比在不同模型上表现可能差异大

### 方案 C：Token 预算驱动
- 特征：不设压缩触发阈值——每次 LLM 调用前自动执行四层上下文治理（Backfill → Microcompact → Budget → Snip），按 token 预算从尾部裁剪历史，边界对齐到 user-turn；治理操作对 LLM 完全透明，在发给 LLM 前已处理完毕
- 优势：避免 big-bang 式压缩的延迟峰值——用渐进式微清理代替一次性大压缩；治理成本平滑化——每轮调用前做轻量清理，而非积累到阈值后大幅压缩；预算上限是精确的 token 计数，行为确定
- 劣势：四层治理逻辑复杂，调试需要逐层排查；Microcompact 将 10 轮次前的 tool result 替换为省略标记，旧信息可能对当前推理有关键意义但已不可用；治理不在 LLM 感知范围内——LLM 不知道哪些内容已被截断或省略

## 设计权衡

**选择硬 token 阈值（方案 A）更合理的场景**：
- 部署环境使用的模型相对固定，context window 大小已知且稳定
- 需要对压缩行为有精确的、跨版本可复现的控制
- Token 预算是成本控制的关键指标，固定阈值与固定预算直接对应
- 压缩阈值与安全边界（如 guard 的 hard limit）需要精确配合

**选择动态窗口百分比（方案 B）更合理的场景**：
- 部署环境可能使用多种不同 context window 大小的模型，或频繁升级模型
- 希望压缩行为与模型能力自然匹配——大窗口模型自动享有更长的未压缩对话
- 压缩的首要目标是避免超限，而非精确的 token 成本控制
- 用户通知阈值需要随模型调整，85%/95% 的语义在任何模型上一致

**关键取舍本质**：方案 A 的阈值是环境的属性——固定、可预测、与模型解耦；方案 B 的阈值是模型能力的函数——自适应、随模型演进、压缩时机随模型能力变化。方案 A 更适合精确控制成本的场景，方案 B 更适合多模型自适应场景。
