# 记忆后端的替换与叠加策略

## 问题陈述

AI Agent 的记忆系统需要持久化存储和语义检索能力。当系统提供内置记忆实现时，外部记忆后端（如专业向量数据库、外部记忆服务）应替换内置实现，还是在内置之上叠加增强？这个选择影响系统的可靠性下限和扩展性上限——替换策略保证架构简洁但可能丢失基础能力，叠加策略保证基础功能始终可用但增加系统复杂度。

## 已知答案图谱

### 方案 A：独占可替换
- 特征：全局只有一个活跃的记忆后端实现，`registerMemoryCapability` 为独占槽位；内置实现（SQLite-vec）和外部实现（LanceDB 等）互斥，用户必须选择一个
- 优势：架构简洁，不存在多后端状态冲突；强迫使用者明确选择一个实现，避免隐式竞争；内置方案保证开箱即用，外部方案支持专业需求
- 劣势：无法组合多个后端的优势（如 SQLite 的零配置 + LanceDB 的高性能）；切换后端时需要迁移数据；不允许同时使用两个后端做不同粒度的记忆

### 方案 B：加性叠加
- 特征：内置记忆始终启用且不可移除，外部 provider 是可选的额外增强（最多 1 个）；BuiltinMemoryProvider 确保基础功能永不依赖外部服务
- 优势：基础功能零配置即可用，不会因为外部服务不可用而导致记忆空白；外部 provider 作为增强层，不影响核心可靠性；内部/外部职责清晰分离（核心记忆 vs 增强记忆）
- 劣势：系统复杂度更高——需要编排两个后端的检索结果、处理结果去重和排序；内存占用更高（两套记忆基础设施同时运行）；配置面上增加了「哪个 provider 管理哪种记忆」的认知负担

## 跨仓库对比

| | OpenClaw | Hermes Agent |
|---|---|---|
| 选择的方案 | 方案 A：独占可替换 | 方案 B：加性叠加 |
| 具体实现 | `registerMemoryCapability` 独占槽位，全局只能有一个活跃实现；内置 backend 使用 SQLite + sqlite-vec 向量扩展；外部实现通过相同接口替换 | MemoryManager 编排 BuiltinMemoryProvider（始终启用 + 不可移除）+ 最多 1 个外部 MemoryProvider（Honcho/Mem0/Supermemory 等 7 种可选）；外部 provider 通过 `config.yaml` 中 `memory.provider` 激活 |
| 付出的代价 | 切换后端时需数据迁移；无法在同一部署中同时使用两种记忆后端的能力 | 两套基础设施的内存和运维成本；检索结果需去重和排序；外部 provider 可用性不影响核心功能但也无法关闭内置以减少资源占用 |

## 设计权衡

**选择独占可替换（方案 A）更合理的场景**：
- 部署环境对资源占用敏感（如边缘设备、个人工具），不希望同时运行两套记忆基础设施
- 记忆后端的差异足够大（如本地文件 vs 云服务），同时使用无意义
- 用户有能力评估和选择最适合的记忆后端，且愿意承担切换成本

**选择加性叠加（方案 B）更合理的场景**：
- 系统面向的用户群体技术水平差异大，必须保证零配置下的基础功能完整
- 外部记忆服务可能不稳定或不可用（如依赖第三方 API），需要内置兜底
- 内部和外部记忆承担不同角色（如内部存短期事实、外部存长期向量记忆），叠加有明确的分工价值

**关键取舍本质**：方案 A 把「选哪个记忆后端」的决策完全交给部署者，追求架构的极致简洁；方案 B 把「保证记忆功能永远可用」作为不可妥协的底线，接受由此带来的复杂度代价。方案 A 更适合有明确后端偏好的技术用户，方案 B 更适合需要零配置开箱即用的大众用户。
