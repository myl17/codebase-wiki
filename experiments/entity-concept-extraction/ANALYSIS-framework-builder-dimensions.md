# Framework Builder 视角：提取维度应该是什么？

> 日期：2026-06-20
> 核心问题：提取设计子问题的维度应该从数据聚类产生，还是从 Framework Builder 的认知需求产生？

---

## 一、Framework Builder 在做什么？

区分两个角色：

| | Framework Builder | Architect |
|---|---|---|
| 输入 | 多个仓库，无特定项目约束 | 一个具体项目 + 一组明确约束 |
| 活动 | 广度优先研究——了解一个类别中所有框架的设计空间 | 约束匹配——在已知选项中找最符合项目需求的 |
| 产出 | 知识储备：设计空间地图、权衡对比表 | 选型决策：选 X 不选 Y，附理由 |
| 时序 | Architect 的上游——为未来的决策做储备 | Framework Builder 的下游——从储备中提取决策依据 |

Framework Builder 的认知任务不是 "哪个最好"，而是 **"哪些设计维度上存在不同选择，每个选择的代价是什么"**。他不急于做决定，但他需要保证——当决定时刻到来时——他没有遗漏重要的设计维度，也没有误解某个框架在关键维度上的实际位置。

---

## 二、正方：Framework Builder 有共通的认知结构

### 论据 1：任务结构是通用的

不管研究的是 AI agent 框架、前端框架、还是数据库，Framework Builder 每次面对一个新框架时问的都是同一组元问题：

1. **这个框架管什么、不管什么？** —— 能力边界
2. **我能改什么、改不了什么？** —— 扩展表面
3. **用它的持续成本是什么？** —— 运维负担
4. **它绑定了什么外部依赖？** —— 锁定风险
5. **它怎么防止我做蠢事？** —— 安全网

这些元问题是跨领域不变的——研究 React 和研究 PostgreSQL 都会问这些问题，只是具体的内容不同。

### 论据 2：遗漏的代价比噪声的代价大

Framework Builder 最怕的不是信息太多，而是**漏了一个关键维度**。如果他研究了 10 个框架的扩展机制、资源策略、安全模型，但漏了"这个框架在并发模型上的假设"——到了选型时，他选的框架可能是单线程的，而他的项目需要多线程。此时他对扩展机制的全部知识都帮不了他。

这意味着：**提取维度宁可多覆盖、少量冗余，也绝不能遗漏。**

### 论据 3：客观的设计空间存在

不同框架在同一个能力域下确实做出了不同的选择——这些差异记录在源码中（interface 定义、配置项、注册机制），不是观察者构造出来的。虽然两个 Framework Builder 可能用不同的标签描述同一个差异，但差异本身是客观的。这意味着**存在一组维度可以系统性地覆盖这些客观差异**。

---

## 三、反方：Framework Builder 的关注点可能严重分化

### 论据 1：背景偏见

- 一个从安全背景来的 Framework Builder 会把 60% 的注意力放在"这个框架怎么防止误操作"上
- 一个从平台工程背景来的 Builder 会把 60% 的注意力放在"这个框架怎么嵌入更大的系统"
- 一个 solo developer 出身的 Builder 会关心"学习曲线和上手成本"——而源码里根本提取不出这个信息

同一组源码证据，不同 Builder 提取出的"重要维度"不同。

### 论据 2：框架类别决定相关维度

- 研究 **前端框架** 时，"渲染策略"（CSR/SSR/SSG/ISR）是核心维度。AI agent 框架完全没有这个维度。
- 研究 **数据库** 时，"一致性模型"和"持久化保证"是核心维度。前端框架完全不涉及。
- 研究 **消息队列** 时，"投递语义"（at-most-once / at-least-once / exactly-once）是核心维度。

如果我声称"扩展边界、资源策略、安全模型这三个维度足够"，那只是因为这三者恰好覆盖了 AI agent 框架的设计空间。换一个框架类别，维度结构可能完全不同。

### 论据 3：Framework Builder 自己都不知道他需要什么

一个 Builder 在研究的早期阶段，他不知道自己会忽略什么。他可能花了三周研究扩展机制，然后才发现他关心的项目场景需要的是"完全不同的扩展粒度"——而他的研究根本没覆盖这个粒度。**Builder 的认知局限恰恰是提取系统应该弥补的。**

## 四、综合：这对维度选择意味着什么

### 4.1 维度来源应该是什么？

| 来源 | 优势 | 劣势 |
|------|------|------|
| **纯数据聚类**（我之前做的） | 保证维度与实际设计差异对齐；不引入人的偏见 | 受限于当前仓库类别；可能遗漏 Builder 关心但数据覆盖不到的维度 |
| **纯 Builder 需求**（自上而下） | 保证回答 Builder 真正的问题 | Builder 自己可能有盲区；可能问出源码无法回答的问题 |
| **数据 + Builder 需求交集** | 既对齐源码证据、又对齐 Builder 问题 | 交集可能太小，遗漏重要维度 |

我认为正确答案是 **数据 + Builder 需求的交集，但对交集之外的区域分开处理**：

```
数据有、Builder 也问的 → 核心提取维度（高置信）
数据有、Builder 不问的 → 潜在盲区补充（告诉 Builder "你可能没注意到这个"）
数据没有、Builder 问的 → 诚实标注边界（"源码只能回答到这里，其余需外部信息"）
```

### 4.2 三个数据派生维度在 Builder 视角下的检验

| 数据派生维度 | Builder 问的元问题 | 源码可回答？ | 结论 |
|-------------|-------------------|------------|------|
| 扩展边界设计 | "这个框架什么可替换、怎么替换？" | ✅（interface、registry、config 均为结构化信号） | 保留 |
| 有限资源策略 | "持续运营的成本是什么？" | ⚠️ 部分（token 策略、延迟特征可推断；服务器成本不可） | 保留但标注局限 |
| 安全与信任模型 | "怎么防止我或 LLM 做蠢事？" | ✅（拦截层位置、审批逻辑、兜底策略均在源码中） | 保留 |

三个维度都通过了 Builder 视角的检验。

### 4.3 但缺了什么？

对照 Builder 的五个元问题，三个维度覆盖了其中三个：

| Builder 元问题 | 是否被三个维度覆盖？ |
|---------------|-------------------|
| 这个框架管什么、不管什么？ | ❌ 不在此层——这是 Ontology 层（能力域清单）的职责 |
| 我能改什么、改不了什么？ | ✅ 维度 1：扩展边界设计 |
| 用它的持续成本是什么？ | ✅ 维度 2：有限资源策略 |
| 它绑定了什么外部依赖？ | ⚠️ 部分——依赖/lock-in 没有独立维度，混在维度 1（make vs delegate）里 |
| 它怎么防止我做蠢事？ | ✅ 维度 3：安全与信任模型 |

有一个潜在的第四维度：**依赖与锁定（Dependency & Lock-in）**。

当前它在维度 1 中以 "make vs delegate" 的形式出现——但 make vs delegate 问的是"这个能力是自己做还是委托出去"，而 lock-in 问的是更硬的约束："这个框架只能在 Linux 上跑"、"这个框架强制使用 PostgreSQL"、"这个框架的所有扩展必须用 TypeScript 写"。**

这些硬约束不在"扩展边界设计"的范围内——它们不是设计选择，而是**不能改变的边界条件**。

### 4.4 是否需要第四维度？

让我用实际数据检验。回头看 17 个 Concept：

- `llm-provider-integration`（make vs delegate）→ 维度 1 已经覆盖
- `execution-engine-decoupling`（执行引擎与产品层的边界）→ 维度 1 覆盖
- 有没有哪个 Concept 是关于"硬约束"的？→ **没有。** 

这意味着两个可能：
1. 三个 AI agent 框架之间在这个维度上没有显著差异（都是 Python/TypeScript，都是跨平台，都没有锁定特定数据库）
2. 当前的提取流程**漏掉了**这个维度

我认为是 1——在当前这批仓库中，确实没有显著的 lock-in 差异。但这不是说维度不该存在——而是说当前的仓库样本不足以检验它。当仓库类型扩展后，这个维度应该被追加。

## 五、修订后的方案

### 5.1 Phase 3a 核心提取维度（三个，有数据支撑且 Builder 关心）

**维度 1：什么可被替换，如何替换？**
```
Builder 对应问题："这个框架的哪些部分我可以换成自己的实现？替换的成本是什么？"
检查清单：
- 哪些部分有 interface/ABC + ≥2 个实现？（扩展点）
- 新实现如何被发现和注册？
- 核心组件允许第三方替换吗？安全守卫是什么？
- 多个扩展的贡献如何组合？
- 这个能力是自己实现还是委托外部？（make vs delegate）
```

**维度 2：有限资源如何分配？**
```
Builder 对应问题："运行这个框架的持续成本是什么？哪些操作会消耗稀缺资源？"
检查清单：
- context window 有限——什么内容占用、什么被裁掉、按什么优先级？
- API 调用有延迟和成本——什么预取、什么按需、什么缓存？
- 压缩/清理是 LLM 驱动（昂贵但智能）还是规则驱动（便宜但机械）？
- 写入操作是同步阻塞还是异步 fire-and-forget？
```

**维度 3：如何防止产生伤害？**
```
Builder 对应问题："这个框架在多大程度上保护我不踩坑？保护机制放在哪一层？"
检查清单：
- 什么操作被认为是危险的？
- 安全措施放在管线的哪一层？
- 安全决策谁做？
- 安全机制失败时如何兜底？
```

### 5.2 第四维度：依赖与锁定（保留，待数据验证）

**维度 4（provisional）：硬约束与锁定风险**
```
Builder 对应问题："如果我选了这个框架，我被锁定了什么？什么是我不能改的？"
检查清单：
- 语言/运行时锁定：只能用某种语言写扩展吗？
- 平台锁定：只能在特定 OS/云上运行吗？
- 数据存储锁定：强制使用特定数据库吗？
- 协议锁定：与其他系统通信的协议是否标准？
```

这个维度在当前 agent 框架样本中可能没有判别力差异（都类似），所以暂时标注为 **provisional——检查它，但如果所有仓库在该维度上没有差异，跳过。** 当仓库类型扩展到包含有显著 lock-in 差异的框架时，升级为正式维度。

### 5.3 跨维度补充：设计哲学（不在此层）

设计哲学（"为什么这个框架反复做出同一类选择"）是 Framework Builder 的合法需求——但它不是 per-Ontology-node 的分析，而是跨节点的综合。应该放在 Phase 3a 完成之后，作为独立的 Synthesis 步骤：

```
Phase 3a: 对每个 Ontology 节点，逐维度提取设计子问题
Phase 3b: 验证
Phase 3c (new): 跨 Ontology 节点综合——识别反复出现的模式 = 设计哲学
```

### 5.4 对于 Framework Builder 背景差异的处理

核心提取维度是**固定的**（保证覆盖面，不被 Builder 个人背景影响），但 Knowledge View 层允许 Builder 按自己关心的轴重新组织：

```
提取层（固定维度）          →  保障不遗漏
展示层（Builder 自选轴）    →  适配不同关注点
```

- 安全背景的 Builder 可以只看维度 3 的产出
- 平台工程的 Builder 可以看维度 1 + 维度 4
- 如果你关心的轴在当前维度中不存在 → 这是信号，说明需要追加维度

---

## 六、总结

1. **Framework Builder 和 Architect 需要区分。** Builder 是广度优先的知识储备，Architect 是约束驱动的决策。维度设计应该服务于 Builder 的元问题："这个框架空间有哪些不同的设计选择？"

2. **三个数据派生维度（扩展边界、资源策略、安全模型）通过了 Builder 视角的检验**——每个都对应一个 Builder 真正会问的元问题，且都能从源码中提取。

3. **三个维度不是完整的。** Builder 还关心"硬约束与锁定"（第四维度），以及"设计哲学"（跨节点综合）。前者应该作为 provisional 维度加入 Phase 3a（当前样本可能无差异，但后续样本会需要），后者应该作为独立的 Phase 3c 步骤。

4. **Framework Builder 的个人背景确实会影响他关注什么**——解决方案不是让提取维度变成可选的（那会导致遗漏），而是提取层固定保证覆盖面，展示层灵活适配不同 Builder 的关注点。

5. **这些维度的最终验证不在理论推导——在增量数据。** 当第 4、第 5 个仓库加入后，如果出现了一个重要设计差异无法归入任何现有维度，那就是追加新维度的信号。反之，如果一个维度在连续 N 个仓库中都没有产生任何判別力差异，那就是削减维度的信号。
