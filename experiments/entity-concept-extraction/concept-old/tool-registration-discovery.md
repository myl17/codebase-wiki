# 工具注册与发现方式

## 问题陈述

AI Agent 框架需要让系统知道有哪些工具可用。核心问题是：新工具如何被系统发现和注册——需要开发者显式调用注册 API 手动声明每个工具，还是系统通过静态分析自动扫描代码中的注册调用？这个选择直接影响添加新工具的摩擦系数、工具清单的可审计性，以及注册机制的隐式约定复杂度。

## 已知答案图谱

### 方案 A：显式 API 注册
- 特征：开发者必须主动调用注册 API（如 `api.registerTool(...)`），每个工具在代码中显式声明其 schema、handler 和元数据；工具清单由所有注册调用共同构成，不依赖静态分析或约定
- 优势：注册行为完全显式，工具清单可审计性最高——grep 注册调用即可获得完整列表；不依赖 AST 扫描或文件约定，新人上手无需理解隐式规则；IDE 类型检查和自动补全对注册 API 参数有效
- 劣势：每添加一个工具都需要写注册样板代码；工具数量增长后注册调用分散在各处，需要额外规范约定注册位置
- **实例**：OpenClaw — `OpenClawPluginApi` 提供 25 个注册方法，每个工具通过 `api.registerTool(...)` 显式声明；nanobot — `registry.register()` 在所有工具被 `_register_default_tools()` 逐一显式点名，每个新工具必须在此出现

### 方案 B：AST 自动发现
- 特征：系统在启动时扫描所有工具模块的 AST，自动识别顶层 `registry.register()` 调用并提取参数（name、toolset、schema、handler）；开发者只需在 Python 文件顶层调用注册函数，无需额外配置或导入中央清单
- 优势：添加新工具的摩擦极低——创建新文件并在顶层调用注册即可，无需修改任何现有文件；注册行为即代码，不会出现「写了工具但忘了注册」的遗漏；toolset 分组和 includes 组合与自动发现无缝配合
- 劣势：依赖静态分析约定（只扫描顶层调用），有一定的隐式假设，新开发者可能不知道这个规则；注册调用的参数错误在启动时才能发现（而非 IDE 类型检查阶段）；AST 扫描逻辑本身是维护负担

## 跨仓库对比

| | OpenClaw | Hermes Agent | nanobot |
|---|---|---|---|---|
| 选择的方案 | 方案 A：显式 API 注册 | 方案 B：AST 自动发现 | 方案 A：显式 API 注册 |
| 具体实现 | `OpenClawPluginApi` 提供 25 个注册方法（`registerContextEngine`、`registerMemoryCapability` 及各类 hook/tool 注册），plugin 在入口文件中命令式调用 API 注册所有能力 | `ToolRegistry` 单例在启动时扫描所有工具模块的 AST，识别顶层的 `registry.register(name=..., toolset=..., schema=..., handler=...)` 调用并自动收集；MCP server 动态刷新时同名工具由 MCP 覆盖 | `_register_default_tools()` 在 AgentLoop.__init__() 中逐一列出所有内建工具，每个新工具必须在此显式点名；MCP 工具以 `mcp_` 前缀动态注入；专用 Schema 类型（StringSchema/IntegerSchema/ObjectSchema 等子类，非泛用 JSON Schema 库） |
| 付出的代价 | plugin 作者必须理解 25 个注册方法的用途和调用时机；注册样板代码量随工具数量线性增长 | AST 扫描只识别顶层调用，嵌套或条件注册被忽略；注册参数错误在运行时才暴露；隐式约定需要文档和开发者共识 | 新增工具需手动注册——不能创建文件即自动发现；注册集中在 AgentLoop 构造函数中，该函数随工具增长而膨胀 |

## 设计权衡

**选择显式 API 注册（方案 A）更合理的场景**：
- Plugin/扩展生态需要严格的接口契约，注册 API 本身就是 contract 的一部分
- 注册行为需要 IDE 类型检查和编译时验证（TypeScript 优势）
- 工具数量相对较少，注册样板代码不是主要负担

**选择 AST 自动发现（方案 B）更合理的场景**：
- 工具数量多且频繁增减，手动维护注册清单会成为瓶颈
- 希望「添加文件即添加工具」的零摩擦体验
- 运行在动态语言环境（Python），AST 扫描是天然可用的能力
- MCP 等动态协议需要运行时刷新工具清单，自动发现比手动维护更容易适应

**关键取舍本质**：方案 A 将注册定义为显式的 API 契约——可审计、可类型检查，但需要开发者主动维护；方案 B 将注册定义为隐式的代码约定——零摩擦、自发现，但依赖静态分析的正确性和开发者的约定遵守。
