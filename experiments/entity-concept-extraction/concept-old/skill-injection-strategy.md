# 技能/知识注入的加载策略

## 问题陈述

AI Agent 框架通常以 Markdown 技能文件（SKILL.md）作为知识注入的主要载体。核心问题是：这些技能内容应该在 prompt 组装时全量注入 system prompt（让 LLM 在一次调用中看到所有可用知识），还是渐进式按需加载（system prompt 中只放技能摘要，LLM 需要时通过工具按需读取完整内容）？这个选择直接影响 system prompt 的 token 消耗、技能数量的可扩展性，以及 LLM 发现和使用技能的效率。

## 已知答案图谱

### 方案 A：全量注入
- 特征：所有可用技能在 system prompt 组装时全文注入——LLM 在一次调用中看到所有技能的全部内容，无需额外的文件读取工具调用即可使用任何技能
- 优势：LLM 可以一次性理解所有可用技能的全貌——不需要「猜测」某技能是否相关后去读取；消除了按需加载的延迟——技能内容立即可用；技能数量的上限受 context window 限制，但小型技能集合下完全没有问题
- 劣势：技能数量增长后 system prompt 膨胀——每个技能的全文都占据 token 空间，无论本轮对话是否使用；prompt cache 的稳定前缀被技能全文化削弱——新增或修改一个技能会导致整个 system prompt 改变；技能内容不能被 LLM 选择性地忽略

### 方案 B：渐进式按需加载
- 特征：system prompt 中只包含技能摘要（名称、描述、路径、可用性），LLM 发现需要的技能后通过 `read_file` 工具按需读取完整内容；同时支持少数高频技能标记为 `always` 全文注入
- 优势：system prompt 保持精简——只包含核心身份和技能摘要，不论技能数量如何增长；prompt cache 稳定性高——skill 文件的修改不影响 system prompt 前缀（只有摘要格式变化才影响）；可无限扩展技能数量——每个技能只在被需要时消耗一次文件读取的 token 数
- 劣势：LLM 每次使用技能需要额外的工具调用——增加了单次任务的轮次和延迟；LLM 可能遗漏摘要中看似不相关但实际有用的技能——摘要信息不足以准确判断相关性；按需加载依赖 LLM 的自觉性——LLM 需要主动决定「我应该读取这个技能」而不被提示

### 方案 C：混合策略（Always 全文 + 摘要 + 按需读取 + 依赖过滤）
- 特征：核心技能标记 `always: true` 全文注入；其余技能以 XML 摘要注入（名称、描述、路径、可用性）；LLM 通过 `read_file` 按需读取摘要中感兴趣的技能；`requires` 依赖检查（`requires.bins`、`requires.env`）自动过滤掉环境不满足的技能
- 优势：在方案 B 基础上增加了两重改进——(1) Always 技能保证了核心能力始终可用，不依赖 LLM 的判断；(2) 依赖过滤保证了不可用技能不会出现在摘要中误导 LLM；Always 和摘要的分界让开发者可以主动控制哪些技能「值得」全文注入
- 劣势：增加了技能配置维度（`always` 和 `requires`），开发者需要额外判断哪些技能应该标记为 always；依赖检查只覆盖显式声明的依赖，隐式依赖仍可能让技能不可用但出现在摘要中

## 跨仓库对比

| 选择的方案 | 方案 A：全量注入 | 方案 A：全量注入 | 方案 C：混合策略 |
| 具体实现 | Skills 作为纯 Markdown 文件，通过 hook 在 prompt 组装时全文注入 system prompt；所有技能在每次 LLM 调用时全部可见 | 技能以 SKILL.md 文件存储，每次会话启动时全文注入 system prompt；不使用向量检索或按需加载 | 双路径：`always: true` 标记的技能全文注入；其余技能以 XML 摘要形式注入（名称 + 描述 + 路径 + 可用性）；agent 通过 `read_file` 工具按需读取完整技能；frontmatter 中的 `requires.bins` 和 `requires.env` 自动过滤不可用技能 |

## 设计权衡

**选择全量注入（方案 A）更合理的场景**：
- 技能数量少且稳定（通常 < 20 个），system prompt 的 token 消耗完全可控
- 每次对话可能用到多种技能，预先注入比多次读取更高效
- 不希望对 LLM 的「技能发现」增加任何摩擦——所有技能始终可见

**选择渐进式加载（方案 B）更合理的场景**：
- 技能数量可能持续增长，但每次对话实际使用的技能很少
- Prompt cache 的稳定性是硬需求——修改一个技能不应影响所有对话的缓存命中
- 接受额外的工具调用轮次作为按需加载的成本

**选择混合策略（方案 C）更合理的场景**：
- 少数核心技能需要保证始终可用（Always），大量辅助技能需要可扩展（摘要 + 按需）
- 技能存在环境依赖（需要特定二进制或环境变量），需要在注入前自动过滤
- 希望在 token 效率和技能覆盖率之间取得平衡——核心能力全量、长尾能力按需

**关键取舍本质**：方案 A 将 token 预算预先分配给了所有技能——无论是否使用，知识已经在那里，代价是 system prompt 长度随技能数量线性增长；方案 B 将 token 预算按需分配给实际使用的技能——system prompt 保持精简，代价是 LLM 需要额外的工具调用轮次来加载技能；方案 C 将技能分为两个等级——Always 技能享受方案 A 的即时可用性，其余技能享受方案 B 的可扩展性，代价是增加了配置决策的负担。
